# 签到计划奖励拆分重构设计

## 1. 背景

签到模块的接口已经调整，计划基础信息与奖励配置不再通过同一个接口保存。

当前前端实现仍然存在以下问题：

- 「基础信息」与「基础奖励」仍在同一表单中编辑。
- 保存时仍将基础奖励拼入计划创建、更新接口。
- 奖励配置缺少独立交互，无法直观配置按周、按月的日历奖励。
- 连续签到奖励没有面向业务的可视化编辑能力。

本次设计目标是在不修改 Vben 基座的前提下，仅在业务层完成签到计划弹窗和奖励配置的重构，使页面交互与最新接口职责保持一致。

## 2. 设计目标

### 2.1 目标

- 将「基础信息」与「奖励配置」拆分为同一弹窗内的两个步骤。
- 第一步只维护计划基础信息，不再出现基础奖励字段。
- 第二步集中配置默认基础奖励、按日奖励、连续签到奖励。
- 奖励配置必须以直观的日历形式呈现。
- 周计划与月计划都要基于真实时间语义进行配置。
- 提交时按接口职责顺序保存：先计划，再奖励。

### 2.2 非目标

- 不修改 `packages/**`、`internal/**`、`scripts/**` 等基座代码。
- 不调整签到管理页签结构，不新增独立页面路由。
- 不手工修改生成的 `apps/web-ele/src/api/**` 与 `apps/web-ele/src/api/types/**` 文件。
- 不顺带重构奖励对账页签。

## 3. 现状事实

### 3.1 当前接口事实

当前生成的签到接口已经拆分为两组：

- 计划接口：
  - `checkInPlanCreateApi`
  - `checkInPlanUpdateApi`
  - `checkInPlanDetailApi`
  - `checkInPlanUpdateStatusApi`
- 奖励接口：
  - `checkInPlanRewardConfigCreateApi`
  - `checkInPlanRewardConfigUpdateApi`

计划创建、更新接口只包含以下字段：

- `planName`
- `planCode`
- `status`
- `cycleType`
- `allowMakeupCountPerCycle`
- `startDate`
- `endDate`

奖励接口当前负责以下数据：

- `baseRewardConfig`
- `dailyRewardRules`
- `streakRewardRules`

### 3.2 当前页面事实

当前签到管理页位于 `apps/web-ele/src/views/user-manager/check-in/`。

现状实现存在以下耦合：

- `model/shared.ts` 同时承担表单 schema、奖励解析、表格列、提交 payload 构造。
- `index.vue` 直接使用通用 `EsModalForm` 打开签到计划编辑弹窗。
- 通用 `EsModalForm` 适合常规表单，不适合承载多步骤流程、日历配置区、连续奖励卡片区。

## 4. 用户已确认的业务决策

以下结论已经在设计阶段与需求方确认：

- 同一弹窗内完成计划配置，不拆成两个独立弹窗。
- 弹窗必须分上下步骤，不能把基础信息和奖励配置挤在同一视口里。
- 默认基础奖励仍然保留，但必须从基础信息中拆出，移动到奖励配置步骤。
- 连续签到奖励使用卡片列表交互，不使用时间轴或混排复杂布局。
- 月周期必须按照计划实际的起止年月展示。
- 月周期展示采用「单月切换视图」，通过上一月、下一月切换。
- 编辑已有计划时，如果当前没有奖励配置，奖励保存走 `create`；如果已有奖励配置，走 `update`。
- 第一步点击「下一步」时只做前端校验，不落库。
- 点击最终「完成并保存」时，再按顺序提交计划接口与奖励接口。

## 5. 交互设计

### 5.1 弹窗总览

新增、编辑签到计划统一改为签到模块专用弹窗组件，不再复用通用 `EsModalForm`。

弹窗分为两个步骤：

1. `基础信息`
2. `奖励配置`

顶部使用步骤指示器，底部按钮根据步骤切换：

- 第一步：
  - `取消`
  - `下一步`
- 第二步：
  - `上一步`
  - `完成并保存`

### 5.2 第一步：基础信息

第一步只保留计划本身字段：

- 计划名称
- 计划编码
- 计划状态
- 周期类型
- 每周期补签次数
- 开始日期
- 结束日期

以下字段从第一步彻底移除：

- 基础奖励积分
- 基础奖励经验
- 任何按日奖励或连续奖励相关字段

点击「下一步」时仅做前端校验，不触发接口请求。

### 5.3 第二步：奖励配置

第二步只负责奖励配置，页面分为左右两栏：

- 左侧固定摘要区
- 右侧配置工作区

#### 左侧摘要区

左侧显示只读的计划摘要：

- 计划名称
- 计划编码
- 周期类型
- 生效时间窗
- 每周期补签次数

摘要区底部放置「默认基础奖励」编辑区：

- 积分输入框
- 经验输入框
- 辅助说明：未单独配置日期时回退到默认基础奖励

#### 右侧工作区

右侧包含两个固定区域：

- 上半区：按日奖励日历
- 下半区：连续奖励卡片列表

### 5.4 按日奖励交互

#### 周计划

周计划固定展示 7 个日期格：

- 周一
- 周二
- 周三
- 周四
- 周五
- 周六
- 周日

每个格子展示当前状态：

- 已单独配置奖励：显示奖励摘要
- 未单独配置：显示「沿用默认」

点击某个格子后，在日历下方展示编辑区：

- 当前编辑日期
- 积分输入框
- 经验输入框

如果两个值都清空，则表示该日期不再单独配置，回退为默认奖励。

#### 月计划

月计划必须按计划真实起止年月展示，不允许抽象成固定 `1-31` 配置板。

展示规则如下：

- 一次只显示一个自然月。
- 顶部提供 `上一月` / `下一月` 切换器。
- 当前月份标题显示为 `YYYY 年 M 月`。
- 开始月中，早于计划开始日期的格子置灰且不可编辑。
- 结束月中，晚于计划结束日期的格子置灰且不可编辑。
- 时间窗之间的完整月份按正常自然月展示。
- 非当前月份补位日期置灰显示，不可编辑。

月计划日历中的每个真实日期格，同样显示：

- 已单独配置奖励
- 沿用默认

点击某个可编辑日期后，在日历下方展示编辑区，行为与周计划一致。

### 5.5 连续奖励交互

连续奖励区域顶部放置显式按钮：

- `新增连续奖励`

点击后立即在列表顶部插入一张空白规则卡，不开启二级弹窗。

每张规则卡内提供以下编辑项：

- 连续天数
- 积分
- 经验
- 是否可重复
- 规则状态

每张卡还需要提供：

- 删除按钮
- 清晰的状态标签

新建卡片完成编辑后，列表按 `连续天数` 升序重排，以便快速理解奖励阶梯。

## 6. 数据模型设计

### 6.1 前端表单模型拆分

当前单一表单模型拆为两个业务模型：

#### 计划基础信息模型

- `id`
- `planName`
- `planCode`
- `status`
- `cycleType`
- `allowMakeupCountPerCycle`
- `startDate`
- `endDate`

#### 奖励配置模型

- `baseRewardPoints`
- `baseRewardExperience`
- `dailyRewardRules`
- `streakRewardRules`
- `currentMonthCursor`（月计划浏览中的当前月份）
- `selectedDayKey`（当前编辑日期）

#### 按日奖励规则前端结构

按日奖励建议在前端统一归一为显式结构，避免直接操作接口中的 `any[]`：

```ts
type CheckInDailyRewardDraft = {
  dayKey: string;
  dayIndex: number;
  experience?: number;
  points?: number;
};
```

说明：

- 周计划中，`dayKey` 可使用固定语义键，如 `weekly-1` 至 `weekly-7`。
- 月计划中，`dayKey` 使用真实日期字符串，如 `2026-05-03`。
- 提交到接口前再转换为后端所需的 `dayIndex + rewardConfig` 结构。

#### 连续奖励规则前端结构

```ts
type CheckInStreakRewardDraft = {
  localId: string;
  streakDays: number;
  points?: number;
  experience?: number;
  repeatable: boolean;
  status: 0 | 1;
};
```

`localId` 仅用于前端渲染与新增卡片跟踪，不提交给接口。

### 6.2 详情回填策略

编辑场景下，前端先调用 `checkInPlanDetailApi`，将返回数据拆分为：

- 计划基础信息草稿
- 奖励配置草稿

拆分规则：

- `baseRewardConfig` 映射到默认基础奖励输入区
- `dailyRewardRules` 映射到日历数据源
- `streakRewardRules` 映射到连续奖励卡片列表

## 7. 提交设计

### 7.1 提交时机

整个弹窗只有点击最终「完成并保存」时才真正发起请求。

第 1 步与第 2 步之间只做本地状态切换，不触发接口。

### 7.2 提交顺序

点击最终保存后，固定按以下顺序执行：

1. 提交计划基础信息
2. 提交奖励配置

#### 新增场景

1. 调用 `checkInPlanCreateApi`
2. 成功后使用计划标识调用 `checkInPlanRewardConfigCreateApi`

#### 编辑场景

1. 调用 `checkInPlanUpdateApi`
2. 根据奖励配置现状判断：
   - 已存在奖励配置：调用 `checkInPlanRewardConfigUpdateApi`
   - 不存在奖励配置：调用 `checkInPlanRewardConfigCreateApi`

### 7.3 奖励配置是否存在的判断

前端通过详情接口的真实返回值判断奖励配置是否存在，判定依据为：

- 存在 `baseRewardConfig`
- 或存在 `dailyRewardRules`
- 或存在 `streakRewardRules`

只要任一项有实际配置内容，即视为奖励配置已存在，保存时走 `update`。

如果三项都没有有效内容，则视为尚未创建奖励配置，保存时走 `create`。

## 8. 校验规则

### 8.1 基础信息校验

沿用现有业务规则：

- 计划名称必填
- 计划编码必填
- 状态必填
- 周期类型必填
- 每周期补签次数必填
- 开始日期必填
- 结束日期不能早于开始日期
- 周计划开始日期必须为周一
- 周计划结束日期必须为周日
- 月计划开始日期必须为每月 1 号
- 月计划结束日期必须为月末

### 8.2 默认基础奖励校验

默认基础奖励积分与经验至少填写一项正整数。

如果产品后续允许「无默认基础奖励」，则需要单独调整该规则；本次实现按当前已确认需求执行，不留模糊分支。

### 8.3 按日奖励校验

- 每个单独配置的日期必须至少填写一项正整数奖励。
- 清空某日期的积分与经验，等价于删除该日期单独配置，回退为默认基础奖励。
- 月计划中不可配置时间窗外日期。
- 月计划中不可配置非当前自然月的补位日期。

### 8.4 连续奖励校验

- 连续天数必填。
- 奖励至少填写一项正整数。
- 同一计划内连续天数不得重复。
- 连续天数必须大于 0。

## 9. 错误处理

### 9.1 计划接口失败

如果计划保存失败：

- 停留在当前弹窗，不关闭
- 不继续请求奖励接口
- 在弹窗顶部或当前步骤顶部显示错误提示

### 9.2 奖励接口失败

如果计划保存成功但奖励保存失败：

- 保留弹窗打开状态
- 明确提示「计划已保存，但奖励配置保存失败」
- 保留当前奖励草稿，允许用户修正后再次提交

### 9.3 表单级错误表达

复杂映射逻辑与错误分支需要补充简洁注释，说明：

- 草稿结构与接口结构的转换关系
- 月计划时间窗裁剪规则
- 编辑态下 `create/update` 选择逻辑

## 10. 页面结构与代码拆分

### 10.1 组件拆分

新增签到专用弹窗组件，例如：

- `apps/web-ele/src/views/user-manager/check-in/components/check-in-plan-modal.vue`

该组件负责：

- 两步弹窗状态切换
- 基础信息表单
- 奖励配置视图
- 提交控制

`index.vue` 只负责：

- 列表页签
- 打开弹窗
- 列表刷新

### 10.2 模型与工具拆分

`model/shared.ts` 需要按职责拆分，至少拆出以下能力：

- 基础信息表单 schema
- 奖励配置草稿模型
- 周计划日历映射工具
- 月计划真实年月日历映射工具
- 奖励提交 payload 构造函数
- 编辑态奖励配置存在性判断函数

如果单文件继续膨胀，可进一步拆分为：

- `model/plan-form.ts`
- `model/reward-form.ts`
- `model/calendar.ts`
- `model/payload.ts`

是否进一步拆文件，以实现时实际体量为准，但必须保持边界清晰。

### 10.3 详情展示同步调整

详情弹层仍保留，但展示逻辑要与新接口语义一致：

- 「基础奖励配置」改为基于奖励配置真实数据展示
- 如果按日奖励存在多档配置，展示摘要而非把整个日历塞入详情卡片
- 连续奖励规则展示数量和核心摘要即可

## 11. 验证策略

本次为业务页面重构，完成后至少执行最小必要验证：

- 与签到页面相关的类型检查
- 页面可用性验证

如果补充自动化测试，优先覆盖纯函数逻辑：

- 月计划真实年月映射
- 周计划日期映射
- 奖励 payload 构造
- 编辑态 `create/update` 判断

不保留临时调试代码与临时测试文件。

## 12. 风险与控制

### 风险 1：奖励接口类型为 `any[]`

风险：

- 前端直接拼装请求时容易出现结构漂移。

控制：

- 在业务层定义显式草稿结构与 payload 转换函数。

### 风险 2：月计划时间窗映射复杂

风险：

- 起始月、结束月裁剪容易出现越界日期或错误可编辑状态。

控制：

- 将月历映射逻辑提炼为纯函数。
- 为映射函数增加最小必要测试或独立验证。

### 风险 3：编辑态奖励配置的 `create/update` 分支判断错误

风险：

- 可能出现保存到错误接口，导致接口报错或覆盖失败。

控制：

- 统一封装存在性判断逻辑。
- 使用详情回填结果作为唯一判断来源。

## 13. 最终结论

本次签到模块重构采用以下方案：

- 同一弹窗内的两步式流程
- 第一步只配置计划基础信息
- 第二步独立配置默认基础奖励、按日奖励、连续奖励
- 周计划按周历配置
- 月计划按真实起止年月的单月切换日历配置
- 连续奖励采用卡片列表原地新增、编辑、删除
- 最终保存时按 `计划接口 -> 奖励接口` 顺序提交
- 编辑态下根据详情中的真实奖励配置是否存在来决定调用 `create` 还是 `update`

该方案与当前接口职责、用户确认的交互偏好、仓库改动边界约束保持一致，可进入实现计划阶段。
