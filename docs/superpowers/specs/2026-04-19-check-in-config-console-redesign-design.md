# 签到配置台重构设计

## 1. 背景

当前 `es-admin` 的签到管理页面仍然延续了旧的“签到计划 + 周期 + 对账”心智，但 `es-server` 的管理端合同已经收敛为新的签到模型：

- 单例全局签到配置
- 单一当前连续奖励轮次
- 奖励对账与补偿

现状错位主要体现在：

- 页面仍以“计划管理”为核心对象
- 旧的 `plan/cycle` 编辑器与当前接口合同不一致
- 奖励编辑依然偏向表单列表，不够贴近运营的“看着日历直接配置”的习惯
- 当前前端内部状态仍然过度依赖“积分/经验两个快捷字段”，不利于未来扩展到更多奖励项

本次设计目标是在**不修改 Vben 基座**、**不扩展本次 `es-server` 的多奖励类型发放能力**的前提下，把签到模块重构为面向真实业务的“配置台”。

## 2. 设计目标

### 2.1 目标

- 将签到管理页重构为三段式配置台：
  - `基础配置`
  - `连续奖励轮次`
  - `奖励对账`
- 移除旧的“计划管理”与 `plan/cycle` 业务心智。
- 将 `基础配置` 的奖励编辑改成“预览驱动”：
  - 运营先看周/月预览
  - 直接点日期格子或周格子打开抽屉编辑
- 当前版本仅开放 `积分 / 经验` 两种奖励输入，但前端底层状态和提交结构向 `rewardItems[]` 靠拢。
- `连续奖励轮次` 只允许编辑当前轮次；历史轮次只读查看。
- 明确对运营解释：
  - 修改当前轮次会产生新的当前版本
  - 老用户继续完成旧轮，完成后再进入新轮
  - 后继链路由系统托管，不由运营人工编排

### 2.2 非目标

- 不改 `packages/**`、`internal/**`、`scripts/**` 等 Vben 基座。
- 本次不扩展 `es-server` 的多奖励类型发放能力。
- 本次不引入未来轮次草稿管理、轮次图编辑或手工后继编排。
- 不再兼容旧“签到计划”页面交互和表单结构。

## 3. 现状事实

### 3.1 `es-server` 合同现状

当前签到管理接口已切换到新模型：

- `checkInConfigDetailApi`
- `checkInConfigUpdateApi`
- `checkInConfigUpdateEnabledApi`
- `checkInStreakRoundDetailApi`
- `checkInStreakRoundUpdateApi`
- `checkInReconciliationPageApi`
- `checkInReconciliationRepairApi`
- `checkInStreakRoundHistoryPageApi`
- `checkInStreakRoundHistoryDetailApi`

这说明后端已经不再以 `plan/cycle` 资源作为管理对象。

### 3.2 `es-admin` 页面现状

当前签到页面仍是旧结构：

- 顶层双 tab：`计划管理 / 奖励对账`
- 依赖旧的计划弹窗、计划详情和周期奖励预览弹窗
- 计划基础信息、奖励配置、连续奖励规则强耦合在旧编辑器中

这与当前后端模型不匹配。

## 4. 用户已确认的业务决策

以下结论已经在本轮设计阶段确认：

- 旧签到模块按新模型重做，不再兼容旧 `plan/cycle` 心智。
- 页面顶层采用三个 tab：
  - `基础配置`
  - `连续奖励轮次`
  - `奖励对账`
- `连续奖励轮次` 需要显式展示链路概念，但高级配置区默认展开，并带说明与风险提示。
- `基础配置` 的奖励编辑采用“混合式”：
  - 默认列表总览
  - 预览卡片直接点击格子进入编辑
- 具体日期奖励和周期模式奖励都应通过预览卡片上的格子直接配置。
- 编辑器使用**抽屉**而不是就地展开或右侧固定面板。
- 本次后端暂不扩展多奖励类型发放能力。
- 但前端内部数据结构要向 `rewardItems[]` 靠拢，避免后续再次推翻。
- 目前运营真实方式是：
  - 只会在需要变更时直接把当前轮改成新版本
  - 老用户继续完成旧轮，再进入新轮
  - 历史轮次只读查看，不做编辑

## 5. 总体方案

本次采用“**预览驱动配置台**”方案。

核心思想：

- `基础配置` 不再以规则列表为主编辑器，而改成“预览即编辑”
- 列表区降级为“规则总览 / 审查区”
- 奖励编辑入口统一收敛到抽屉
- 前端将 `rewardItems[]` 作为真实数据源，`积分 / 经验` 只是当前阶段的快捷编辑方式

## 6. 页面结构设计

### 6.1 顶层结构

页面保留原有路由位置，但整体形态改为三段式配置台：

1. `基础配置`
2. `连续奖励轮次`
3. `奖励对账`

不再出现：

- `计划管理`
- `新增签到计划`
- 任意历史轮次编辑入口
- 任何需要运营手工维护后继图的 UI

### 6.2 `基础配置` Tab

分为三个区域：

#### A. 全局配置区

用于维护：

- 签到功能开关
- 补签周期（周 / 月）
- 每周期补签额度
- 页面级保存按钮

#### B. 奖励预览区（主编辑器）

这是 `基础配置` 的核心区域。

- 周模式：
  - 展示一周 7 天格子
  - 支持点某一天，配置：
    - 仅当前日期奖励
    - 每周这一天奖励
- 月模式：
  - 展示月历
  - 支持点某个日期，配置：
    - 仅当前日期奖励
    - 每月这一天奖励
    - 如果点到月末，可配置“按月最后一天”奖励

每个格子需要展示：

- 当前命中奖励摘要
- 是否沿用默认基础奖励
- 规则作用范围提示

#### C. 规则总览区（辅助区）

保留列表，但不再作为主编辑器。

用途：

- 快速审查当前已配置的日期规则和周期模式规则
- 点击已有规则时，仍然打开同一个编辑抽屉
- 便于运营发现重复、遗漏和冲突

### 6.3 `连续奖励轮次` Tab

页面分为两块：

#### A. 当前轮次编辑区

用于编辑：

- `roundCode`
- 当前轮次奖励规则列表

页面上明确说明：

- 当前只维护一个可编辑轮次
- 保存时会生成新的当前版本
- 老用户完成旧轮后再进入新轮

#### B. 历史轮次只读区

展示：

- 版本号
- 状态
- 前驱 / 后继上下文
- 时间信息

点击后打开只读详情抽屉，展示：

- 奖励规则快照
- 前驱 / 后继关系
- 只读说明

不提供任何编辑动作。

### 6.4 `奖励对账` Tab

尽量保留已有表格式交互：

- 查询
- 状态查看
- 奖励补偿

只移除旧 `plan/cycle` 相关筛选与文案。

## 7. 交互设计

### 7.1 `基础配置` 的核心交互

1. 运营在顶部设置：
   - 开关
   - 周 / 月
   - 补签额度
2. 进入奖励预览区
3. 点击任意格子
4. 打开抽屉编辑
5. 修改奖励
6. 保存到页面本地状态
7. 页面级点击“保存基础配置”时统一提交

### 7.2 抽屉编辑器交互

抽屉标题必须明确当前对象，例如：

- `编辑 2026-04-21 奖励`
- `编辑每周周三奖励`
- `编辑每月 15 号奖励`
- `编辑每月最后一天奖励`

抽屉内分两层：

#### 第一层：快捷编辑区

显示两个快捷字段：

- `积分`
- `经验`

这满足当前后端发奖能力的约束，也符合运营当前使用习惯。

#### 第二层：底层奖励项区

默认折叠，可展开查看 `rewardItems[]`。

当前阶段作用：

- 展示快捷字段映射后的底层结构
- 让前端状态模型完成泛化

本次不要求运营直接维护复杂奖励项，只需要保证内部结构已经过渡到 `rewardItems[]`。

### 7.3 保存策略

采用“**本地缓冲 + 页面级统一保存**”：

- 点击格子后修改，只更新本地状态
- 不在格子级别立即请求后端
- 页面级点击保存后，统一构造 payload 提交

这样可以减少碎片化请求，提升运营编辑体验。

### 7.4 `连续奖励轮次` 交互

- 页面加载即展示当前轮次
- 支持新增 / 编辑 / 删除当前轮次规则
- 高级配置说明区默认展开
- 保存时使用页面级统一提交
- 历史轮次只能点“查看详情”

## 8. 前端数据模型设计

### 8.1 状态拆分

前端状态拆成三个 owner：

1. `config`
2. `round`
3. `reconciliation`

对应文件：

- `model/config.ts`
- `model/streak-round.ts`
- `model/reconciliation.ts`
- `model/shared.ts`

### 8.2 `rewardItems[]` 作为真实数据源

本次关键设计是：

- `rewardItems[]` 是真实数据源
- `points / experience` 只是当前阶段的快捷编辑投影

也就是说：

- 用户改“积分 / 经验”时，内部要同步回写到 `rewardItems[]`
- 后续如果增加道具、虚拟货币等奖励，不需要推翻现有状态模型

### 8.3 `config` 状态

应包含：

- `enabled`
- `makeupPeriodType`
- `periodicAllowance`
- `baseRewardItems`
- `dateRewardRules`
- `patternRewardRules`
- 预览游标
- 当前选中格子
- 当前抽屉编辑目标

### 8.4 抽屉编辑目标

建议统一建模，例如：

- `kind`: `base | date | weekday | monthDay | monthLastDay`
- `targetDate`
- `targetWeekday`
- `rewardItems`
- `displayRewardForm`

这样“点格子编辑”和“点规则列表编辑”可以走同一套逻辑。

### 8.5 `round` 状态

建议保持简单：

- `roundCode`
- `version`
- `status`
- `rewardRules`

提交时统一映射为当前后端 DTO，但前端固定传输值：

- `status = ACTIVE`
- `nextRoundStrategy = INHERIT`
- `nextRoundConfigId` 不传

原因：

- 当前这些字段不是运营可编辑概念
- 它们只是当前后端传输契约的一部分

## 9. 提交流程设计

### 9.1 基础配置提交

点击 `保存基础配置` 时：

1. 从 `config` 本地状态构造 `UpdateCheckInConfigDto`
2. 其中规则数据统一输出为 `rewardItems[]`
3. 调用 `checkInConfigUpdateApi`
4. 成功后重新拉 `checkInConfigDetailApi`

### 9.2 当前轮次提交

点击 `保存当前轮次` 时：

1. 从 `round` 本地状态构造 `UpdateCheckInStreakRoundDto`
2. 固定附带：
   - `status = ACTIVE`
   - `nextRoundStrategy = INHERIT`
   - `nextRoundConfigId` 省略
3. 调用 `checkInStreakRoundUpdateApi`
4. 成功后重新拉当前轮次详情与历史列表

### 9.3 对账提交

保持现有：

- `checkInReconciliationPageApi`
- `checkInReconciliationRepairApi`

不额外扩复杂流程。

## 10. 兼容与扩展策略

### 10.1 当前阶段兼容

虽然前端状态已经泛化到 `rewardItems[]`，但当前后端真实发奖能力仍只有：

- 积分
- 经验

因此本次 UI 只开放这两类快捷编辑。

### 10.2 后续扩展

如果后续 `es-server` 支持更多奖励类型：

- 不需要重写整个签到页面
- 只需要扩展抽屉编辑器
- 让底层 `rewardItems[]` 真正支持更多资产类型

## 11. 验证要求

本次改动完成后，至少需要验证：

### `es-server`

- `pnpm type-check`
- 签到相关 targeted Jest
- `pnpm publish-api:admin`

### `es-admin`

- `pnpm att:ele`
- `pnpm -F @vben/web-ele run typecheck`
- 签到模型层 Vitest
- 搜索确认：
  - 存在 `基础配置 / 连续奖励轮次 / 奖励对账`
  - 不存在 `计划管理 / 新增签到计划 / checkInPlan*`

## 12. 风险与控制

### 风险 1：交互做成“预览即编辑”后逻辑分叉过多

控制：

- 抽屉编辑目标统一建模
- “点格子”和“点规则”共用同一个抽屉

### 风险 2：前端表面泛化，内部仍残留 `points/experience` 写死逻辑

控制：

- 明确 `rewardItems[]` 为真实数据源
- 快捷字段只是映射投影

### 风险 3：运营误解当前轮次的可编辑范围

控制：

- 高级说明区默认展开
- 明确写清：
  - 当前轮次可编辑
  - 历史轮次只读
  - 后继链路由系统托管

## 13. 最终结论

本次签到模块前端重构采用以下方案：

- 页面整体改为三段式配置台
- `基础配置` 采用“预览驱动”交互
- 奖励编辑通过抽屉完成
- 具体日期奖励和周期模式奖励都支持在预览格子上直接点击编辑
- 列表区仅作规则总览和辅助审查
- `连续奖励轮次` 只编辑当前轮次，历史轮次只读
- 前端内部状态向 `rewardItems[]` 过渡
- 当前 UI 仍只开放 `积分 / 经验`

这个方案最符合当前业务、当前后端能力边界，以及后续多奖励项扩展方向。
