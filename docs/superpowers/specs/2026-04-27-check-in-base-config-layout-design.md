# 签到基础配置页布局收口设计

## 1. 背景

当前签到模块的基础配置页，将以下两块内容做成了左右并列：

- 签到基础配置
- 默认基础奖励

这种布局会带来明显的认知偏差：

1. 用户会误以为“默认基础奖励”和“基础配置”是两个同级对象。
2. 右侧卡片内容稀疏，但占据了较强视觉权重。
3. 用户设置完周期、补签额度后，操作重心本应进入奖励预览，而不是跳到一个孤立的默认奖励卡片。

本设计用于收口该页面的布局层级，明确基础配置页的主次关系。

## 2. 设计目标

- 消除“基础配置”和“默认基础奖励”被并列展示带来的信息架构歧义。
- 让页面主流程回到“先配全局参数，再看奖励预览，再补充覆盖规则”。
- 将默认基础奖励明确为奖励体系的默认值，而不是独立一级模块。
- 不改变当前抽屉编辑、统一保存、`rewardItems[]` 作为真实数据源等既有决策。

## 3. 非目标

- 不重写整个签到模块。
- 不改变连续奖励轮次与奖励对账的页面结构。
- 不新增奖励类型，不修改后端合同。
- 不调整 Vben 基座或共享底座代码。

## 4. 已确认决策

本次页面按单主列四段式组织：

1. 全局配置
2. 默认奖励
3. 奖励预览
4. 规则总览

不再保留“签到基础配置卡 + 默认基础奖励卡”的左右并列双卡布局。

## 5. 信息层级原则

页面层级必须明确表达以下关系：

- 第一层：签到系统如何运行
  - 开关
  - 周期
  - 每周期补签额度
  - 补签图标
- 第二层：默认奖励是什么
  - 当没有命中更具体规则时发什么
- 第三层：哪些规则覆盖默认奖励
  - 具体日期奖励
  - 周期模式奖励

默认奖励属于“奖励体系默认值”，不属于和全局配置并列的一级业务对象。

## 6. 页面结构设计

### 6.1 总体结构

页面采用单列布局，自上而下组织为四个区块：

```text
签到基础配置                                   [保存基础配置]

[全局配置]
签到开关 | 签到周期 | 每周期补签额度 | 补签图标

[默认奖励]
积分 12 / 经验 12                        [编辑默认奖励]
未命中具体日期奖励和周期奖励时，发放默认奖励

[奖励预览]
周模式：一周 7 格
月模式：月历格子
点击任意格子 -> 打开抽屉编辑

[规则总览]
周期规则列表
具体日期规则列表
```

### 6.2 页面头部

页面头部保留：

- 标题：`签到基础配置`
- 页面级保存按钮：`保存基础配置`

保存按钮放在头部右侧，而不是散落在局部卡片内部。这样可以明确本页采用“本地缓冲 + 页面统一保存”的交互模型。

### 6.3 全局配置区

全局配置区只承载运行参数，不承担奖励编辑职责。

包含字段：

- 签到开关
- 签到周期
- 每周期补签额度
- 补签图标

布局建议：

- 桌面端使用两列到三列栅格，保证字段密度适中
- 移动或窄宽度下自然折为单列
- 补签图标作为同一区块中的一个字段项，不单独占据大面积留白

### 6.4 默认奖励区

默认奖励区从“独立大卡”改为“摘要型配置条”。

必须包含：

- 奖励摘要，例如：`积分 12 / 经验 12`
- `编辑默认奖励`按钮
- 一行解释文案：未命中具体日期奖励和周期奖励时，发放默认奖励

交互要求：

- 点击编辑按钮，仍然复用当前默认奖励抽屉编辑器
- 不在该区块内展开复杂表单
- 不把该区块设计成与主配置区同等视觉重量的独立面板

视觉要求：

- 该区块应明显可见，但权重低于“奖励预览”
- 采用紧凑摘要表达，避免大片空白

### 6.5 奖励预览区

奖励预览区是本页主编辑器，视觉权重最高。

展示规则：

- 周模式时展示一周 7 个格子
- 月模式时展示月历格子
- 每个格子展示当前命中奖励摘要
- 未配置覆盖规则时，格子明确展示“沿用默认”或等价提示
- 格子需提示当前规则作用范围

交互要求：

- 点击格子后打开统一抽屉编辑器
- 继续沿用当前“本地编辑，页面统一保存”策略
- 默认奖励不在此区直接编辑，但预览区需能清楚体现哪些格子正在继承默认奖励

### 6.6 规则总览区

规则总览区保留，但降级为辅助审查区，不再作为主编辑器。

用途：

- 查看已配置的具体日期规则
- 查看已配置的周期模式规则
- 点击已有规则，打开同一个抽屉编辑器
- 帮助运营发现重复、遗漏和覆盖关系

该区域应放在奖励预览区下方，避免与主编辑路径竞争视觉焦点。

## 7. 推荐交互流

用户在本页的推荐操作顺序为：

1. 设置签到开关、周期、补签额度、补签图标
2. 检查默认奖励是否合理
3. 进入奖励预览区，按日期或周期补充覆盖规则
4. 在规则总览区做快速复查
5. 点击页面级按钮统一保存

这个流程必须通过布局本身被自然表达出来，而不是依赖额外说明文字去补救。

## 8. 对现有实现的调整建议

当前页面中的以下结构需要调整：

- 去掉左右并列双卡布局
- 去掉右侧独立“默认基础奖励”大卡
- 将默认奖励收敛为位于全局配置下方的摘要区
- 将视觉重心转移到奖励预览区
- 将规则列表明确降级为辅助总览

如果实现时需要修改组件结构，优先在业务页面组件内完成，不向底座下沉抽象。

## 9. 验收标准

满足以下条件即可认为本次布局设计达标：

1. 用户不会再把“默认基础奖励”理解为与“基础配置”并列的一级模块。
2. 页面首屏不会出现大面积右侧空卡留白。
3. 用户设置完全局参数后，视觉和操作重心自然流向奖励预览区。
4. 默认奖励能被快速查看和编辑，但不会抢占主编辑器地位。
5. 规则总览仍可用于复查和回跳编辑，但不再承担主编辑职责。
