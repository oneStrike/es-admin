# 注释、表格、表单、弹窗使用规范

## 1. 适用范围

本规范适用于当前仓库中 `apps/web-ele` 的后台页面开发，重点约束以下四类内容：

- 注释与说明文案
- 列表页表格
- 编辑与筛选表单
- 弹窗与详情浮层

本规范基于当前项目已有封装整理而成，优先级如下：

1. 先遵循项目现有抽象层与代码模式
2. 再补充社区通用最佳实践
3. 如业务确有特殊性，可以在评审中说明后例外处理

## 2. 项目基线

当前项目已经形成了较明确的基础约定，新页面默认不要绕过这些封装直接从零开始搭 UI。

### 2.1 表格基线

- 列表页统一优先使用 `useVbenVxeGrid`
- 查询参数统一通过 `formatQuery` 处理分页、排序、筛选
- 搜索区统一通过 `createSearchFormOptions` 生成
- 单元格展示优先复用全局渲染器：`CellDate`、`CellImage`、`CellTag`、`CellText`

### 2.2 表单基线

- 表单统一优先使用 `useVbenForm`
- 组件类型统一走 `#/adapter/component`
- 校验统一优先使用 `zod` 或项目已注册规则：`required`、`selectRequired`、`arrayRequired`、`uploadRequired`
- 搜索表单和编辑表单的字段，优先从同一份 `schema` 演化，而不是分别手写两套含义重复的定义

### 2.3 弹窗基线

- 弹窗统一优先使用 `useVbenModal`
- 表单弹窗优先复用 `EsModalForm`
- 选择类弹窗优先复用 `EsModalTable`
- 只读详情优先复用 `EsRecordDetail`
- 弹窗数据共享统一通过 `modalApi.setData()` / `modalApi.getData()`

### 2.4 当前代码里已经体现出的推荐模式

- 页面级 CRUD：`Grid + FormModal + DetailModal`
- `schema` 放在 `model/shared.ts`
- 详情卡片放在 `model/detail.ts`
- 列表页成功反馈统一是“成功提示 + reload”
- 行内危险操作统一二次确认

## 3. 注释与说明文案规范

这里的“注释”分为两类：

- 代码注释：给开发者看
- 说明文案：给用户看，例如表单 `help`、`description`、空状态、确认文案

### 3.1 代码注释规范

#### 3.1.1 什么时候必须写注释

以下场景必须补充注释：

- 业务规则不是代码表面能直接看懂，例如“为什么只能临时封禁填写截止时间”
- 状态流转存在前置条件、副作用、接口耦合
- 为兼容历史接口、第三方组件、框架限制而做的特殊处理
- 复杂数据转换、字段映射、排序规则、权限判断
- 临时兜底逻辑、已知风险点、后续待替换方案

以下场景不要写注释：

- 变量名和函数名已经足够表达语义
- 注释只是把代码翻译成中文
- 留下一大段已废弃代码当“备份”

#### 3.1.2 注释写法

- 导出的函数、类型、常量、复杂配置块优先使用 `/** */`
- 局部流程说明优先使用简短的 `//`
- 注释重点写“为什么”，少写“做了什么”
- 注释必须和代码一起维护，逻辑变了先改注释再提交
- `TODO`、`FIXME` 必须带上下文，至少写清“要改什么、为什么现在不改”

#### 3.1.3 推荐示例

```ts
/**
 * 搜索表单公共配置
 * 统一控制列表页筛选区的布局、折叠与自动提交行为
 */
export const DEFAULT_SEARCH_FORM_OPTIONS = {
  ...
};

// 超过多选上限时回滚到上一次已确认的选择，避免 UI 与真实状态不一致
function handleSelectionChange(records: any[]) {
  ...
}
```

#### 3.1.4 不推荐示例

```ts
// 定义一个表单
const [Form, formApi] = useVbenModal(...)

// 循环数组
list.forEach(...)
```

### 3.2 用户可见说明文案规范

项目表单 `schema` 已支持 `placeholder`、`help`、`description`，三者职责必须分开：

- `label`：字段名，表达“这是什么”
- `placeholder`：输入示例或动作提示，表达“怎么填”
- `help`：短帮助，表达“特殊规则是什么”
- `description`：较长说明，表达“背景、影响、补充约束”

#### 3.2.1 使用规则

- 不要用 `placeholder` 代替 `label`
- 一句话以内的补充规则优先用 `help`
- 较长说明放 `description`
- 敏感信息、不可逆操作、上传限制必须给解释
- 错误提示要可执行，不要只写“输入有误”

#### 3.2.2 推荐示例

```ts
{
  component: 'DatePicker',
  fieldName: 'banUntil',
  label: '截止时间',
  help: '仅临时禁言和临时封禁需要填写，永久状态可留空',
  componentProps: {
    placeholder: '请选择截止时间',
    type: 'datetime',
    valueFormat: 'YYYY-MM-DD HH:mm:ss',
  },
}
```

## 4. 表格使用规范

### 4.1 默认实现方式

新建列表页时，默认使用下面的结构：

1. `model/shared.ts` 定义列和筛选字段
2. 页面中定义 `gridOptions`
3. 通过 `useVbenVxeGrid({ gridOptions, formOptions })` 挂载
4. 查询接口统一通过 `proxyConfig.ajax.query`
5. 分页与排序统一通过 `formatQuery`

### 4.2 列定义规范

#### 4.2.1 通用要求

- 第一列默认放序号，使用固定左侧
- 主标识列放在左侧优先区域，例如名称、标题、账号
- 操作列固定右侧
- 时间、状态、开关、操作列尽量保持单行
- 长文本统一开启溢出提示，不直接撑坏布局
- 空值统一显示 `-`
- 布尔/枚举值禁止裸展示，必须映射成人类可读文案

#### 4.2.2 列宽建议

- 序号列：`60-80`
- 状态列：`90-120`
- 时间列：`150-180`
- 操作列：按操作数量控制在 `150-260`
- 长文本列：优先 `minWidth` + `showOverflow`

#### 4.2.3 渲染优先级

优先级从高到低如下：

1. 通用渲染器 `CellDate` / `CellImage` / `CellTag` / `CellText`
2. `formatter`
3. 插槽

只有在需要交互、复合布局、特殊展示时才使用插槽，避免所有列都写成模板。

### 4.3 搜索区规范

- 搜索区统一通过 `createSearchFormOptions` 生成
- 搜索字段只保留高频筛选项，不把编辑表单完整复制过来
- 搜索项默认隐藏标签，依赖占位符表达语义
- 搜索项数量超过一行时默认支持折叠
- 常用字段优先放前面：名称、状态、时间区间、业务主键
- 日期范围优先使用范围选择器，不拆成两个分离字段

### 4.4 交互规范

- 行内开关适用于“切换后立即生效”的状态，例如启用/禁用、推荐/不推荐
- 删除、清空、恢复等危险动作必须二次确认
- 行内主操作不超过 3 个，更多操作放到二级入口
- 列表页成功反馈统一用轻提示；失败反馈根据严重程度升级
- 加载、空状态、无权限状态必须可感知

### 4.5 数据与性能规范

- 每个表格必须有稳定 `row key`，当前项目默认是 `id`
- 大列表必须走服务端分页
- 排序默认走远程排序，不在前端伪排序接口数据
- 搜索表单联动变更多时，避免一次交互触发多次重复查询
- 复杂单元格尽量复用渲染器或纯函数，不在插槽里堆过多业务逻辑

### 4.6 推荐模板

```ts
const gridOptions: VxeGridProps<Row> = {
  columns,
  proxyConfig: {
    ajax: {
      query: ({ page, sorts }, formValues) =>
        pageApi(formatQuery({ page, sorts, formValues })),
    },
    sort: true,
  },
};

const [Grid, gridApi] = useVbenVxeGrid({
  gridOptions,
  formOptions: createSearchFormOptions(searchSchema),
});
```

## 5. 表单使用规范

### 5.1 字段来源规范

- 编辑表单字段定义统一放在 `model/shared.ts`
- 同一业务域内，优先维护一份主 `schema`
- 能从主 `schema` 衍生搜索项、表格列时，优先使用 `formSchemaTransform`
- 不允许同一字段在不同文件里维护两套冲突的名称、校验和占位符

### 5.2 控件选择规范

- 单行短文本：`Input`
- 多行内容：`Input + textarea`
- 二选一或少量可比选项：`RadioGroup`
- 多选项：`CheckboxGroup`
- 选项较多或需要搜索：`Select`
- 即时开关：`Switch`
- 时间/日期：`DatePicker`
- 上传：`Upload`
- 富文本：`RichText`
- 远程数据选择：`ApiSelect` / `ApiTreeSelect` / `TableSelect`

补充约束：

- `RadioGroup` 适合少量选项，通常不要超过 5 个
- `Switch` 只用于立即生效的切换，不与“保存”按钮绑在一起表达同一动作
- 上传组件必须说明格式、数量、大小、用途
- 长文本不要硬塞进单行输入框

### 5.3 校验规范

- 必填项必须有前端校验
- 前端校验不能替代后端校验，但要尽量前置发现问题
- 校验文案必须直接指出问题和期望格式
- 正则校验优先配合示例占位符
- 复杂依赖校验优先使用 `zod` 或 `dependencies.rules`

推荐做法：

- 普通必填：`required`
- 选择必填：`selectRequired`
- 上传必填：`uploadRequired`
- 复杂格式、长度、跨字段关系：`z`

### 5.4 布局规范

- 弹窗编辑表单统一使用 `vertical`
- 标准 CRUD 表单优先两列布局
- 单列场景：字段少、字段长、说明多、移动端优先
- 多行文本、富文本、上传、表格选择等宽内容使用整行跨度
- 搜索表单与编辑表单不要共用同一布局

推荐尺寸与列数：

- 简单表单：`cols = 1`，宽度约 `700`
- 标准编辑表单：`cols = 2`，宽度约 `900-1000`
- 复杂选择弹窗：`1200+`
- 复合业务操作台：按内容宽度扩展到 `1400`，但需控制信息分组

### 5.5 文案规范

- `label` 用业务名词，不要用接口字段名
- `placeholder` 用动作语气，例如“请输入昵称”“请选择状态”
- 同一页面同类字段文案必须统一
- 不要出现“请输入内容”“请选择信息”这类无信息量提示

### 5.6 提交与重置规范

- 提交时必须锁定弹窗，防止重复提交
- 成功后统一关闭弹窗并刷新列表
- 失败时保留现场，不自动清空用户输入
- 重置按钮只重置表单，不承担“撤销后端操作”的语义
- 长耗时提交应有 loading 或锁定反馈

推荐写法：

```ts
const [BaseForm, formApi] = useVbenForm({
  layout: 'vertical',
  showDefaultActions: false,
  handleSubmit: async (values) => {
    modalApi.lock();
    try {
      await onSubmit(values);
      modalApi.close();
    } finally {
      modalApi.unlock();
    }
  },
});
```

## 6. 弹窗使用规范

### 6.1 选择弹窗类型

按场景选择，不要一个弹窗包打天下：

- 编辑类：`EsModalForm`
- 选择类：`EsModalTable`
- 详情类：`EsRecordDetail`
- 纯确认类：`el-popconfirm` 或统一确认方法

### 6.2 弹窗标题规范

- 新增：`新增{业务对象}`
- 编辑：`编辑{业务对象}`
- 详情：`{业务对象}详情`
- 选择：`选择{业务对象}`
- 避免使用“弹窗”“窗口”“操作”这类无业务语义标题

### 6.3 打开方式规范

- 外层页面只负责设置数据和打开弹窗
- 弹窗内部负责消费 `sharedData` 并处理自身状态
- 使用 `connectedComponent` 时，不在模板上传一堆 Modal 原生 props
- 需要修改标题、宽度、按钮状态时，优先通过 `useVbenModal` 或 `modalApi.setState()`

推荐写法：

```ts
const [Form, formApi] = useVbenModal({
  connectedComponent: EsModalForm,
});

function openFormModal(record?: Record<string, any>) {
  formApi.setData({
    title: '作者',
    record,
    schema: formSchema,
  }).open();
}
```

### 6.4 关闭与销毁规范

- 纯查看类弹窗可以允许点击遮罩关闭
- 提交中、危险流程、关键确认类弹窗不应允许误触关闭
- `destroyOnClose` 只在确实需要重建内部状态或降低内存占用时开启
- 关闭后若需要清理表单或重置选中态，应在 `onClosed` 中显式处理

### 6.5 嵌套弹窗规范

- 默认不推荐嵌套弹窗
- 能平铺成兄弟弹窗就不要嵌套
- 确有必要时，内层弹窗必须是一个更轻的子任务，例如“选择关联对象”“补充说明”
- 不允许在一个主弹窗里再连续打开多层深弹窗

当前项目中已存在嵌套使用，因此不是绝对禁止，但必须满足：

- 主任务上下文明确
- 不造成焦点混乱
- 不影响关闭路径
- 内层弹窗职责单一

### 6.6 弹窗反馈规范

- 成功且不打断流程的结果，使用轻提示
- 失败但用户可继续修正，优先留在当前弹窗内给出错误
- 重要失败、权限风险、不可逆提醒，升级为对话框或更强提醒
- 危险操作默认二次确认，不把删除直接挂在主按钮上

## 7. 推荐与不推荐清单

### 7.1 推荐

- 使用 `schema` 统一驱动表单、搜索项、部分列表列
- 使用 `help`/`description` 而不是把规则全塞进 placeholder
- 使用统一的成功反馈和刷新逻辑
- 使用固定左右列控制长表格可读性
- 使用项目已有封装组件，而不是在页面里重复造轮子

### 7.2 不推荐

- 直接裸写一套 `el-dialog + el-form + el-table` 绕过抽象层
- 表格列全部用插槽，导致定义分散
- 搜索区和编辑区维护两套完全重复但不一致的字段
- 用 `Switch` 表达“最终保存前才生效”的配置
- 用 placeholder 承担全部说明职责
- 弹窗里堆过多主按钮或危险按钮
- 注释只描述代码表面动作，不描述业务原因

## 8. 页面开发落地顺序

建议按下面顺序开发新页面：

1. 先定义业务对象的主 `schema`
2. 再决定哪些字段进入搜索区
3. 再定义表格列与状态展示方式
4. 最后组合 `Grid / Form / DetailModal`
5. 补齐帮助文案、错误提示、确认文案和必要注释
