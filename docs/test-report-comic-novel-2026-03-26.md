## 4. 错误分类（业务错误 vs 缺陷）

### 4.1 业务预期错误（正常）

1. 前端必填校验拦截
- 现象：空表单提交时报“请上传封面/请输入名称/请选择作者...”等
- 结论：业务规则正确生效

2. 章节号重复冲突
- 接口：`POST /api/api/admin/content/comic/chapter/create`
- 返回：`409`, `message: "该作品下章节号已存在"`
- 结论：业务冲突校验，属于正常业务错误

3. 创建参数缺失校验
- 示例1：`isPublished`缺失 -> `400`, `message: "isPublished必须是布尔类型"`
- 示例2：`tagIds`为空 -> `400`, `message: "tagIds必须是数组类型...数组不能为空"`
- 结论：后端参数校验正常

### 4.2 缺陷（需修复）

#### 缺陷 D1：漫画章节拖拽排序不可用
- 严重级别：高
- 复现步骤：漫画 -> 任意作品 -> 章节管理 -> 拖拽第1行到第2行
- 预期：排序成功并提示“排序调整成功”
- 实际：提示“数据已存在”，排序不生效
- 证据：
  - 请求：`POST /api/api/admin/content/comic/chapter/swap-sort-order`
  - 响应：`409`
  - 返回体：`{"code":409,"data":null,"message":"数据已存在"}`

#### 缺陷 D2：小说章节拖拽排序不可用
- 严重级别：高
- 复现步骤：小说 -> 任意作品 -> 章节管理 -> 拖拽第1行到第2行
- 预期：排序成功并提示“排序调整成功”
- 实际：提示“数据已存在”，排序不生效
- 证据：
  - 请求：`POST /api/api/admin/content/novel/chapter/swap-sort-order`
  - 响应：`409`
  - 返回体：`{"code":409,"data":null,"message":"数据已存在"}`

## 5. 建议

1. 优先修复章节拖拽排序接口（漫画/小说）
- 建议后端在交换序号时使用事务或临时占位序号，避免唯一键冲突

2. 为主体创建冲突返回更具体错误文案
- 当前部分冲突仅返回“数据已存在”，建议细化到字段级（如名称重复/别名重复）

3. 可补充自动化回归
- 将“章节拖拽排序成功”纳入回归用例，防止后续回归失效
